Accessibility isn't optional. I made systematic accessibility improvements across the Supabase dashboard and website.

## Table Editor Accessibility

The Table Editor needed significant a11y work:

- Keyboard navigation for entity items
- Screen reader announcements
- Focus management
- ARIA labels

```tsx title="Accessible entity list"
<EntityListItem
  role="button"
  tabIndex={0}
  aria-label={`Table ${table.name}, ${table.rowCount} rows`}
  onKeyDown={handleKeyDown}
/>
```

## Side Panel Accessibility

Side panels for editing:

- Focus trap within panel
- Escape to close
- Return focus on close
- Screen reader announcements

## Notification Accessibility

Notifications v2 with proper a11y:

- ARIA live regions
- Role="alert" for errors
- Keyboard dismissal
- Focus management

## Form Accessibility

Form components with proper labels:

- Associated labels
- Error messages linked
- Required field indication
- Help text connections

## Dropdown Animations

I made dropdowns animate on keyboard navigation:

```tsx title="Keyboard animation"
<DropdownItem
  onKeyDown={(e) => {
    if (e.key === 'ArrowDown') {
      animateToNext()
    }
  }}
/>
```

## Color Contrast

Improved contrast ratios:

- Primary button updates
- Text on backgrounds
- Focus indicators
- Error states

## RLS Assistant

The RLS assistant with accessibility:

- Template selection keyboard navigable
- Screen reader friendly
- Clear error messaging
- Proper heading hierarchy

## Impact

Accessibility improvements:

1. Enable all developers to use Supabase
2. Improve usability for everyone
3. Meet compliance requirements
4. Demonstrate inclusive values

Accessibility is a feature, not an afterthought.
